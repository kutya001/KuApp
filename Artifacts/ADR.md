## Что такое ADR

Architecture Decision Record — это короткий документ, фиксирующий **важное техническое решение**, принятое в проекте. Каждый ADR содержит контекст, само решение, его последствия и статус.

## ADR-001: Выбор стека — Vanilla JS без сборки

**Статус:** ✅ Принят  
**Дата:** Январь 2024  
**Контекст:** Выбор фронтенд-стека для платформы

### Контекст

Требуется создать мульти-приложение (20+ мини-приложений) с минимальной сложностью поддержки. Рассматривались варианты:

1. **React/Vue SPA** с build-системой (Webpack/Vite)
2. **Vanilla JS** в single-file HTML
3. **Micro-frontends** с отдельными билдами

### Решение

Выбран **Vanilla JS в single-file HTML** без сборщиков и npm-зависимостей.

### Обоснование

| Критерий | Vanilla JS | React + Build |
|----------|-----------|---------------|
| Сложность деплоя | GitHub Pages (статические файлы) | Требует CI/CD для билда |
| Скорость загрузки | Мгновенная (нет бандла) | Зависит от размера бандла |
| Поддержка | Любой текстовый редактор | Нужны Node.js, npm, конфиги |
| Изоляция приложений | Полная (каждый HTML —独立的) | Общая зависимость |
| Порог входа | HTML + JS = готово | Нужно знать фреймворк |

### Последствия

**Положительные:**
- Нулевая сложность деплоя — `git push` и готово
- Каждое приложение полностью изолировано
- Нет зависимостей, которые нужно обновлять
- Любой разработчик может открыть и изменить

**Отрицательные:**
- Нет типизации (TypeScript невозможен без билда)
- Нет tree-shaking, code splitting
- Дублирование кода между приложениями
- Сложное состояние共享 между приложениями только через PlatformAPI

## ADR-002: Бэкенд — Google Sheets через Google Apps Script

**Статус:** ✅ Принят  
**Дата:** Январь 2024  
**Контекст:** Выбор решения для хранения данных

### Контекст

Требуется серверная часть для хранения пользовательских данных, результатов игр и статистики. Бюджет проекта — $0/месяц. Рассматривались:

1. **Firebase** (Firestore, Auth)
2. **Supabase** (PostgreSQL)
3. **Google Sheets** + Google Apps Script
4. **SQLite** + хостинг

### Решение

Выбраны **Google Sheets** с прокси через Google Apps Script.

### Обоснование

| Критерий | Google Sheets | Firebase | Supabase |
|----------|--------------|----------|----------|
| Стоимость | Бесплатно | Бесплатно (лимиты) | Бесплатно (лимиты) |
| Настройка | 5 минут | 30 минут | 1 час |
| Визуальный доступ к данным | Да (открыть таблицу) | Нет (нужен консоль) | Нет (нужен SQL клиент) |
| Лимиты | 10M ячеек, 20k запросов/день | 50k чтений/день | 500MB БД |
| Интеграция с фронтендом | Простая (HTTP POST) | SDK | REST API |

### Последствия

**Положительные:**
- Полностью бесплатно
- Данные видны «глазами» — легко дебажить
- Нет отдельного сервера для поддержки
- CRUD из коробки

**Отрицательные:**
- Rate limiting (~100 запросов/100 сек)
- Нет транзакций — возможны race conditions
- Нет индексов — фильтрация на клиенте
- Нет внешних ключей — целостность на приложении
- Макс. 10M ячеек на таблицу
- Время выполнения GAS — 6 минут

## ADR-003: Архитектура Platform — iframe + PlatformAPI

**Статус:** ✅ Принят  
**Дата:** Февраль 2024  
**Контекст:** Как изолировать и связать 20+ приложений

### Контекст

Каждое приложение должно быть:
- Изолировано от других (CSS, JS глобальные переменные)
- Способно взаимодействовать с платформой (аутентификация, БД, отправка результатов)
- Загружаться и выгружаться без перезагрузки страницы

Рассматривались:

1. **iframe** с postMessage API
2. **Web Components** с Shadow DOM
3. **Динамическая загрузка** скриптов в общий DOM

### Решение

Каждое приложение загружается в **`<iframe>`**, связь через `window.parent.PlatformAPI`.

### Обоснование

| Критерий | iframe | Web Components | Dynamic Scripts |
|----------|--------|---------------|-----------------|
| CSS изоляция | Полная | Shadow DOM | Нет |
| JS изоляция | Полная | Нет | Нет |
| Связь с платформой | Через API | Через события | Через глобальные |
| Простота добавления | HTML файл | Компонент | JS файл |

### Последствия

**Положительные:**
- Полная CSS/JS изоляция каждого приложения
- Приложения не конфликтуют между собой
- Простой API-контракт через PlatformAPI
- Каждое приложение — автономный HTML файл

**Отрицательные:**
- Оверхед iframe (память, производительность)
- Ограниченная связь (только через postMessage/API)
- Нельзя шарить состояние между приложениями напрямую

## ADR-004: PWA с Service Worker для кэширования

**Статус:** ✅ Принят  
**Дата:** Март 2024  
**Контекст:** Обеспечение оффлайн-доступа и установка на устройства

### Контекст

Приложение должно:
- Работать как нативное (без адресной строки)
- Быть устанавливаемым на домашний экран
- Загружаться быстро, даже при плохом интернете

Рассматривались:

1. **Service Worker** с кэшированием
2. **Без PWA** — просто адаптивный сайт
3. **Native app** (React Native, Flutter)

### Решение

Реализован **PWA** с Service Worker (`sw.js`) и стратегиями кэширования.

### Стратегии кэширования

| Ресурс | Стратегия | Причина |
|--------|-----------|---------|
| App shell (index.html, manifest, иконки) | Cache-First | Критично для запуска |
| Файлы приложений (apps/*.html) | Network-First | Могут обновляться |
| Workout JSON (workouts/*.json) | Network-First | Могут обновляться |
| Внешние API | Bypass | Не кэшировать |

### Последствия

**Положительные:**
- Мгновенная загрузка app shell из кэша
- Установка на мобильные устройства
- Работа при плохом интернете
- Offline fallback

**Отрицательные:**
- Нужно вручную управлять версией кэша (`CACHE_NAME`)
- После обновления — требуется перезагрузка
- Данные всё равно требуют сети

## ADR-005: Аутентификация — кастомная через Google Sheets

**Статус:** ✅ Принят  
**Дата:** Февраль 2024  
**Контекст:** Как управлять пользователями и авторизацией

### Контекст

Требуется система аутентификации с ролями (admin/user). Рассматривались:

1. **Firebase Auth**
2. **OAuth** (Google, GitHub)
3. **Кастомная** через Google Sheets

### Решение

**Кастомная аутентификация** с хранением пользователей в `sys_Users` таблице Google Sheets.

### Обоснование

- Firebase Auth требует Firebase проект (дополнительная настройка)
- OAuth требует серверную часть для verification
- Кастомная аутентификация использует уже выбранный бэкенд (Google Sheets)
- Простота: username + password → проверка в таблице

### Последствия

**Положительные:**
- Нет внешних зависимостей
- Полный контроль над данными пользователей
- Легко добавить кастомные поля

**Отрицательные:**
- Пароли хранятся в Google Sheets (риск безопасности)
- Нет recovery через email
- Нет 2FA
- Rate limiting при частых запросах

## ADR-006: TrainingProgramBuilder — React 18 через CDN

**Статус:** ✅ Принят  
**Дата:** Март 2024  
**Контекст:** Выбор стека для конструктора тренировочных программ

### Контекст

Конструктор программ требует:
- Drag-and-drop интерфейса
- Сложного состояния (упражнения, протокол, тайминги)
- JSX для удобной разметки

Рассматривались:

1. **Vanilla JS** с ручной реализацией drag-and-drop
2. **React 18 через CDN** + Babel standalone
3. **Vue 3 через CDN**

### Решение

**React 18 + Babel** загружаются через CDN только для этого приложения.

### Обоснование

| Критерий | Vanilla | React CDN | Vue CDN |
|----------|---------|-----------|---------|
| Drag-and-drop | Ручная реализация | react-dnd / dnd-kit | vuedraggable |
| Управление состоянием | Ручное | useState/useReducer | Reactive |
| JSX | Нет | Да (Babel) | Нет (templates) |
| Размер CDN | 0 | ~150 KB | ~100 KB |

### Последствия

**Положительные:**
- Удобная реализация drag-and-drop
- Декларативное управление состоянием
- JSX для читаемой разметки

**Отрицательные:**
- Единственное приложение с внешними зависимостями
- Загрузка React/Babel замедляет первый запуск
- Babel компиляция в runtime (оверхед)

## ADR-007: Формат тренировочных программ — JSON

**Статус:** ✅ Принят  
**Дата:** Февраль 2024  
**Контекст:** Как хранить и описывать тренировочные программы

### Контекст

Тренировочные программы должны:
- Легко добавляться (без изменения кода)
- Содержать упражнения, протокол, тайминги
- Загружаться приложением «Тренер»

Рассматривались:

1. **JSON файлы** в репозитории
2. **Google Sheets таблица** с программами
3. **Hardcoded** в HTML тренера

### Решение

**JSON файлы** в папке `workouts/` с каталогом в `workouts/index.json`.

### Обоснование

- JSON — стандартный формат, легко читать и редактировать
- Файлы в репозитории — версионируются
- Каталог (`index.json`) позволяет динамически находить программы
- Не требует изменения кода для добавления программы

### Последствия

**Положительные:**
- Добавление программы = создание JSON файла
- Версионирование через git
- Легко тестировать (валидация JSON)

**Отрицательные:**
- Нет UI для создания (кроме TrainingProgramBuilder)
- JSON файлы нужно деплоить с кодом

## ADR-008: Тёмная тема по умолчанию

**Статус:** ✅ Принят  
**Дата:** Январь 2024  
**Контекст:** Выбор цветовой темы платформы

### Контекст

Платформа включает игры и фитнес-приложения. Целевая аудитория — молодые пользователи. Рассматривались:

1. **Светлая тема**
2. **Тёмная тема**
3. **Системная** (автоопределение)

### Решение

**Тёмная тема** по умолчанию (`#0a0a0f` основной фон).

### Обоснование

- Игровые приложения лучше выглядят на тёмном фоне
- Неоновые акценты более выразительны
- Меньше нагрузка на глаза при вечернем использовании
- Стандарт для игровых/фитнес платформ

### Последствия

**Положительные:**
- Эстетичный «arcade» вид
- Лучше для OLED экранов (энергосбережение)
- Меньше усталость глаз

**Отрицательные:**
- Не все пользователи предпочитают тёмную тему
- Нет переключения темы (пока)

## ADR-009: Хостинг — GitHub Pages

**Статус:** ✅ Принят  
**Дата:** Январь 2024  
**Контекст:** Выбор платформы для хостинга

### Контекст

Проект — статические файлы. Требуется бесплатный хостинг с HTTPS. Рассматривались:

1. **GitHub Pages**
2. **Netlify**
3. **Vercel**
4. **Cloudflare Pages**

### Решение

**GitHub Pages** — хостинг из ветки `main`.

### Обоснование

| Критерий | GitHub Pages | Netlify | Vercel |
|----------|-------------|---------|--------|
| Стоимость | Бесплатно | Бесплатно (лимиты) | Бесплатно (лимиты) |
| Настройка | 0 (уже на GitHub) | Нужен аккаунт | Нужен аккаунт |
| CI/CD | `git push` → deploy | Auto или ручной | Auto или ручной |
| Custom domain | Да | Да | Да |
| HTTPS | Да | Да | Да |

### Последствия

**Положительные:**
- Деплой = `git push origin main`
- Бесплатно навсегда
- HTTPS из коробки

**Отрицательные:**
- Нет серверной части (только статика)
- Нет serverless functions
- Нет встроенного CI/CD (но можно добавить Actions)

## ADR-010: Русский язык интерфейса

**Статус:** ✅ Принят  
**Дата:** Январь 2024  
**Контекст:** Выбор языка интерфейса

### Контекст

Основная целевая аудитория — русскоязычные пользователи.

### Решение

**Русский язык** по умолчанию. i18n не реализован.

### Последствия

**Положительные:**
- Проще разработка (нет i18n фреймворка)
- Нет дублирования строк

**Отрицательные:**
- Нельзя использовать англоязычным пользователям
- Добавление другого языка потребует рефакторинга
